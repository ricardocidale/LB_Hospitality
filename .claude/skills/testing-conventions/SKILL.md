Testing conventions and patterns for the financial verification test suite. Covers golden tests, proof scenarios, reconciliation tests, the fixture factory, artifact generation, severity-based reporting, and the `npm run verify` multi-phase pipeline. Use this skill when writing new tests, understanding test patterns, or debugging test failures.

## Test Pattern Taxonomy

### 1. Golden Test Pattern

Hand-calculated reference values with 0% growth/inflation for traceability. Every golden test:

- Uses fixed, deterministic inputs (0% ADR growth, flat occupancy, 0% inflation)
- Derives expected values step-by-step from first principles using named constants
- Never hardcodes expected values — computes them from `shared/constants.ts` exports
- Uses `toBeCloseTo(expected, 2)` for penny-level precision
- Verifies month-over-month invariants (e.g., revenue identical every month when growth is 0%)

**Canonical example:** `tests/golden/full-statement-golden.test.ts`

```typescript
// CORRECT — derive from constants
const H_REV_ROOMS = H_SOLD_ROOMS * 200;
const H_REV_EVENTS = H_REV_ROOMS * DEFAULT_REV_SHARE_EVENTS;
expect(m0.revenueEvents).toBeCloseTo(H_REV_EVENTS, 2);

// WRONG — hardcoded expected value
expect(m0.revenueEvents).toBeCloseTo(25620, 2);
```

**Directory:** `tests/golden/` (13 files, 269 tests)

### 2. Proof Scenario Pattern

Structural scenarios that exercise every financial path through the engine. Five scenarios cover the full matrix:

| # | Scenario | What It Proves |
|---|----------|----------------|
| 1 | Cash purchase | Pure equity, no loan paths |
| 2 | Financed purchase | Debt sizing, amortization, DSCR |
| 3 | Cash → refinance | Refi mechanics, loan swap |
| 4 | Portfolio aggregate | Multi-property, fee linkage |
| 5 | Consolidated + eliminations | Intercompany elimination |

Each scenario validates GAAP identities:
- A = L + E (ASC 210) every month
- OCF = NI + Depreciation (ASC 230)
- CFF = -Principal + Refi Proceeds
- Fee linkage: Σ(SPV fees) = OpCo revenue

**File:** `tests/proof/scenarios.test.ts`

### 3. Reconciliation & Consistency Tests

Generate artifact reports (JSON + Markdown) that prove financial bridges balance:

- **Sources & Uses:** equity + debt = purchase price
- **NOI → FCF bridge:** ANOI - debt service - tax + refi = FCF
- **Cash bridge:** beginning + OCF + FCF = ending
- **Debt reconciliation:** beginning + draws - payments = ending

**Severity categories:**
| Severity | Meaning | Impact on Audit Opinion |
|----------|---------|------------------------|
| `critical` | Calculation error, identity violation | → ADVERSE |
| `material` | Significant discrepancy (>1% tolerance) | → QUALIFIED |
| `warning` | Out-of-range assumption, unusual value | None (informational) |
| `info` | Negative cash balance, business condition | None |

**File:** `tests/proof/reconciliation-report.test.ts`

### 4. Hardcoded Detection

Scans finance source files for magic numbers. Only safe numbers (0, 1, -1, 2, 12, 100) and context exceptions (loop counters, array indices, string literals) are allowed.

**File:** `tests/proof/hardcoded-detection.test.ts`

### 5. Data Integrity Tests

Database-level checks that validate ownership model and uniqueness:
- Exactly one shared `global_assumptions` row (userId IS NULL)
- All properties have userId = NULL (shared across users)
- No duplicate property names
- No orphaned duplicate rows

**File:** `tests/proof/data-integrity.test.ts`

## Test Fixture Factory

All engine tests use a shared fixture factory in `tests/fixtures/index.ts`:

```typescript
import { makeProperty, makeGlobal, baseProperty, baseGlobal } from "../fixtures";

// Override specific fields
const prop = makeProperty({ startAdr: 300, roomCount: 15 });
const global = makeGlobal({ projectionYears: 5 });
```

**Base property:** 10-room Full Equity, $200 ADR, 60% starting occupancy, $1M purchase price.

**Rules:**
- Always use `makeProperty()` / `makeGlobal()` for test inputs — never construct raw objects
- Override only the fields relevant to the test scenario
- The base fixtures match the canonical golden scenario in `tests/engine/proforma-golden.test.ts`

## Artifact Generation

Tests that produce reconciliation reports write to `test-artifacts/`:
- `{scenario-name}.json` — structured reconciliation data
- `{scenario-name}.md` — human-readable Markdown report
- `consolidated-eliminations.json` / `.md` — intercompany elimination report

Artifacts are generated by `tests/proof/reconciliation-report.test.ts` during the verify pipeline. The verify runner's final phase summarizes all artifact opinions.

## `npm run verify` Pipeline

The verification pipeline runs 8 phases sequentially via `tests/proof/verify-runner.ts`:

| Phase | Test File | What It Checks |
|-------|-----------|----------------|
| 1 | `scenarios.test.ts` | 5 golden scenario GAAP identities |
| 2 | `hardcoded-detection.test.ts` | No magic numbers in finance files |
| 3 | `golden-values.test.ts` | Penny-exact hand-calculated values |
| 4 | `reconciliation-report.test.ts` | Artifact generation + bridge checks |
| 5 | `data-integrity.test.ts` | Database ownership + uniqueness |
| 6 | `portfolio-dynamics.test.ts` | Dynamic property count, fee zero-sum |
| 7 | `recalculation-enforcement.test.ts` | All mutations invalidate queries |
| 8 | `rule-compliance.test.ts` | Admin config literals, constants parity |

**Final phase:** Artifact Summary — reads all JSON artifacts and reports opinions.

**Commands:**
```bash
npm run verify          # Full pipeline (UNQUALIFIED = pass)
npm test                # All ~3,000 tests
npx vitest run tests/golden/  # 269 golden tests only (~3s)
```

**Only UNQUALIFIED audit opinion is acceptable.** Any critical or material issue fails the pipeline.

## Rules for Writing New Tests

1. New financial calculations require new tests in the appropriate `tests/` subdirectory
2. New calculators require golden tests with hand-derived expected values
3. Never inline magic numbers — import from `shared/constants.ts`
4. Use the fixture factory (`makeProperty`, `makeGlobal`) for all engine inputs
5. Financial tests verify against engine outputs, not hardcoded expected values
6. All tests must pass before merge: `npm test`
7. Finance changes must produce UNQUALIFIED: `npm run verify`
